# 就活スケジュール管理アプリ 設計書

作成日: 2026-06-08
ステータス: 設計承認済み（実装計画の作成へ）

## 1. 概要

就活のインターン・面接・説明会などの予定を一元管理し、**予定の「被り」を信号機カラーで可視化**して防ぐWebアプリ。さらに、面接で**実際にされた質問**を記録し、**うまく答えられなかった質問を全企業横断でリスト化**することで「次の面接に活かす」ことを目的とする。

大学の期末課題として、**DB必須・Docker必須**の要件を満たす構成で開発する。

## 2. ターゲット / 動機

- 就活中の学生（マルチユーザー、各自が登録・ログインして自分の就活を管理）
- 「インターンや面接の日程が被る」という実体験ベースの課題を解決する
- 単なるカレンダーではなく、被り検出と面接振り返りで差別化する

## 3. 技術スタック

| レイヤ | 採用技術 |
|--------|----------|
| 言語 / フレームワーク | PHP / Laravel |
| DB | MySQL |
| 認証 | Laravel Breeze |
| 画面 | Laravel Blade（サーバーサイドレンダリング）＋ Tailwind CSS ＋ 少量のJavaScript |
| カレンダーUI | FullCalendar.js |
| インフラ | Docker（Laravel Sail：app / mysql コンテナ） |

アーキテクチャ方針: **Bladeによるモノリス構成**（API+SPAは期末課題には過剰なため不採用）。

### フロントエンドの作り方

別フロント（React等）は作らず、**Laravel内でBlade＋Tailwind CSS**で画面を構築する（SSR）。
処理の流れは `ルーティング(routes/web.php) → コントローラー → モデル(Eloquent/MySQL) → ビュー(Blade)` のMVC。

- **認証 / 企業 / 振り返り / 苦手質問リスト画面**: 通常のBladeフォーム・一覧で実装。
- **カレンダー画面（メイン）**: SSRのHTMLだけでは動きが不便なため、**FullCalendar.js** を利用。
  - LaravelがDBから取得した予定を**JSON**でFullCalendarに渡して月/週ビューを描画。
  - 予定の色に**信号機カラー（🔴重複 / 🟡間隔狭 / 通常）**を割り当てる。
  - 予定クリックで詳細・編集へ遷移。

## 4. 機能一覧

1. **認証**（Laravel Breeze）: ユーザー登録・ログイン・ログアウト。データはユーザーごとに完全分離。
2. **企業管理**: 応募先企業をCRUD（企業名・選考状況・メモ）。
3. **予定管理（Googleカレンダー風UI）**: 月/週ビューで予定を登録・編集・削除。
4. **被り検出（信号機カラー）**: 🔴 時間重複 / 🟡 予定間隔が短い（移動リスク） を色で表示。
5. **面接の振り返り**: 各面接予定に「実際の質問 ＋ 手応え ＋ 次はこう答えるメモ」を記録。
6. **苦手質問リスト**: 「うまく答えられなかった質問」を全企業横断で一覧表示し、次の面接前の見直しに使う。

## 5. データモデル

### users（Breeze標準）
- id, name, email, password, timestamps

### companies

### events
| カラム | 型 | 備考 |
|--------|----|----|
| id | bigint PK | |
| user_id | bigint FK → users | 所有者 |
| company_id | bigint FK → companies, nullable | 紐づく企業（任意）|
| title | string | 予定タイトル |

| end_at | datetime | 終了日時 |

### interview_questions
| カラム | 型 | 備考 |
|--------|----|----|
| id | bigint PK | |
| user_id | bigint FK → users | 所有者 |
| event_id | bigint FK → events | 紐づく面接予定 |

| timestamps | | |

リレーション概要:
- User 1—n Company / Event / InterviewQuestion
- Company 1—n Event
- Event 1—n InterviewQuestion

## 6. 被り検出ロジック（信号機カラー）

同一ユーザーの予定同士を比較して各予定に色を付与する。

- **🔴 赤（重複）**: 時間が重なる予定が存在する
  - 判定: `A.start_at < B.end_at AND B.start_at < A.end_at`
- **🟡 黄（要注意 / 移動リスク）**: 重なってはいないが、前の予定の終了から次の予定の開始までの間隔が閾値（既定 **60分**）未満
  - 判定: `0 <= (next.start_at - prev.end_at) < 60分`
- **通常表示**: 上記いずれにも該当しない

優先順位: 赤 > 黄 > 通常（重複があれば黄判定より赤を優先）。

## 7. 画面構成

- **ログイン / 登録**（Breeze標準画面ベース）
- **カレンダー画面（メイン）**: 月/週ビュー。予定を信号機カラーで表示。クリックで予定詳細・編集。
- **予定登録 / 編集フォーム**: タイトル・種別・企業・開始/終了・場所。
- **企業一覧 / 詳細**: CRUD。詳細から関連予定を確認。
- **面接振り返り画面**: 面接予定に対し質問・手応え・改善メモを追加/編集。
- **苦手質問リスト画面**: `result = bad` の質問を企業名・面接日とともに横断一覧表示。

## 8. インフラ / Docker

- Laravel Sail を利用し、`docker-compose.yml` で **app（PHP/Laravel）** と **mysql** の2コンテナ構成。
- `.env` でDB接続情報を管理。
- マイグレーション・シーダーでDB初期化を再現可能にする。

## 9. スコープ外（YAGNI）

期末課題の期間内で確実に完成させるため、以下は今回は作らない:
- メール/プッシュによるリマインド通知（外部インフラが必要なため）
- Googleカレンダー等の外部カレンダー連携
- よく聞かれる質問のランキング/同一質問の自動集計（質問の同一判定が困難なため）
- スコア推移のグラフ可視化（必要になれば後続フェーズで追加検討）

## 10. 成功基準

- マルチユーザーでログインし、自分の予定・企業・質問が他人に見えないこと
- カレンダー上で重複（赤）・間隔狭(黄)が正しく色分けされること
- 面接ごとに質問を記録でき、「答えられなかった質問」が横断リストに出ること
- `docker compose up`（または Sail）でDB込みで起動・動作すること
